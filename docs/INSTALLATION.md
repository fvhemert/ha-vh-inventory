# VH-Inventory — Installation Guide

This guide explains how to install the **VH-Inventory** inventory & grocery system on a
**fresh (virgin) Home Assistant server**. It covers every dependency: the pyscript
integration, the custom Lovelace cards, the backend app, the configuration package, the
SQLite database, and the dashboard builder.

> **Audience:** a Home Assistant administrator with file-system access to `/config`.
> **Time required:** ~30–45 minutes on a clean install.

---

## 1. Architecture overview

VH-Inventory has three layers:

| Layer | What it is | Where it lives |
|---|---|---|
| **Data** | A single SQLite database with all tables (products, stock, locations, categories, stores, shopping list, scan queue, history). | `/config/vh_inventory.db` |
| **Backend** | A [pyscript](https://github.com/custom-components/pyscript) app that creates the database, exposes each table as a `sensor.vh_inventory_*` entity, and provides all read/write services. Plus a configuration *package* of input helpers and scripts. | `/config/pyscript/apps/vh_inventory/__init__.py` and `/config/packages/vh_inventory.yaml` |
| **Frontend** | A tabbed Lovelace dashboard generated by a local Python script and pushed to HA over WebSocket. Includes a live JavaScript translation layer for multi-language UI. | URL path `/vh-inventory`, built by `mk_dash.py` |

```
 Barcode scan / manual input
            │
            ▼
 input helpers + scripts (package YAML)
            │  call services
            ▼
 pyscript app  ──►  SQLite DB (/config/vh_inventory.db)
            │  publishes
            ▼
 sensor.vh_inventory_*  ──►  Lovelace dashboard (flex-table + bubble-card popups)
```

## 2. Prerequisites

- **Home Assistant** (Container, OS, or Supervised) — a recent version with the default
  dashboard and **HACS** installed. If HACS is not yet installed, follow the
  [official HACS setup](https://www.hacs.xyz/docs/use/) first.
- **File access to `/config`** — via the *File editor* / *Studio Code Server* add-on,
  *Samba* share, or *Terminal* add-on.
- **A workstation with Python 3.8+** (Windows/macOS/Linux) to run the dashboard builder.
  The builder uses the `websocket-client` package.
- **A Long-Lived Access Token** from your HA user profile (used by the dashboard builder).
- *(Optional — for printing)* An **ESC/POS thermal printer** (developed against an
  **Epson TM-T20II**) reachable from Home Assistant, plus the **ESC/POS Printer**
  integration that provides the `escpos_printer.print_text` service. Only needed if you want
  the Shopping-list and Inventory **Print** features.

## 3. Install the pyscript integration

The backend runs on **pyscript**.

1. In Home Assistant, open **HACS → Integrations → (⋮) → Custom… / Search** and install
   **Pyscript Python scripting**.
2. **Restart Home Assistant** (Settings → System → Restart).
3. Add pyscript to your configuration. In `configuration.yaml` (or a dedicated file),
   register the app and allow external imports — the app uses the `requests`
   library for online barcode lookups:

   ```yaml
   # configuration.yaml
   pyscript:
     allow_all_imports: true
     apps:
       vh_inventory:      # REQUIRED - see note below
   ```

   > **⚠️ The `apps: vh_inventory:` entry is required.** pyscript loads an app from
   > `/config/pyscript/apps/<name>/` **only if that app has a matching entry under
   > `apps:`**. An empty `apps:` (or omitting the entry) means the backend never loads —
   > no `sensor.vh_inventory_*` entities and no database are created, with no error in the
   > log. The entry can be empty (as above) for a keyless install; add the optional
   > `upcdb_*` keys under it only to enable upcdatabase.org (see §9).

   > **Why `allow_all_imports`?** The backend imports `requests` to query the
   > Open Food Facts / UPC Item DB APIs. Without this flag pyscript blocks the import.

4. Make sure the package include exists in `configuration.yaml` so the YAML package in
   step 5 is loaded. All of the solution's own configuration (input helpers, scripts,
   template sensors) lives in the **dedicated package file** `packages/vh_inventory.yaml` —
   `configuration.yaml` only needs to *link* to it:

   ```yaml
   # configuration.yaml
   homeassistant:
     packages:
       vh_inventory: !include packages/vh_inventory.yaml
       # or load every file in /config/packages: packages: !include_dir_named packages
   ```

   > A complete, copy-ready reference of every `configuration.yaml` addition this solution
   > needs (pyscript, the package link, the recorder change below, and the optional theme
   > loader) is provided in **`configuration.example.yaml`** at the repository root. Merge
   > those blocks into your own `configuration.yaml` — nothing solution-specific is defined
   > inline; it only links to the dedicated files.

5. **Exclude the carrier sensors from the recorder.** Every `sensor.vh_inventory_*` entity
   holds its full table as a state *attribute* (the History sensor alone is ~30 KB), and the
   backend rewrites all of them on every mutation. The SQLite database (`vh_inventory.db`) is
   the source of truth, so recording their state history has no value and would bloat the
   recorder DB (and trip Home Assistant's 16 KB attribute warning).

   **This is the one system-level change the solution requires in `configuration.yaml`.**
   `recorder` is a single-instance integration, so it **must** be a top-level key in
   `configuration.yaml` and **cannot** be placed inside the package file. If you already have
   a `recorder:` block, **merge** the `exclude` below into it rather than adding a second
   `recorder:` key. Add it and **restart** — recorder config is only read at startup, there
   is no live reload:

   ```yaml
   # configuration.yaml
   recorder:
     exclude:
       entity_globs:
         - sensor.vh_inventory_*
   ```

   To clear any history recorded before the exclusion was added, call the
   `recorder.purge_entities` service with `entity_globs: ["sensor.vh_inventory_*"]` and
   `keep_days: 0` (otherwise old rows age out at the normal retention period).

## 4. Install the custom Lovelace cards

The dashboard is built from several HACS **frontend** cards. Install each from
**HACS → Frontend**:

| Card | `type:` used | Purpose |
|---|---|---|
| **button-card** | `custom:button-card` | Action buttons, hidden boot card for the JS layer |
| **card-mod** | (global styling) | Conditional styling (e.g. red cart when shopping list has items) |
| **flex-table-card** | `custom:flex-table-card` | All data tables (stock, products, scan queue, history…) |
| **bubble-card** | `custom:bubble-card` | Add/Edit pop-up dialogs |
| **tabbed-card** (programmable) | `custom:tabbed-card-programmable` | The tab bar across the top of the dashboard |
| **kiosk-mode** | (global resource) | Hides the HA header when the *Hide header (kiosk)* toggle is on |

> ⚠️ **Note on `tabbed-card-programmable`:** this is a *programmable* variant of the
> tabbed-card resource. Install the tabbed-card resource that provides the
> `custom:tabbed-card-programmable` element. If your HACS copy registers a different
> element name, either install the matching variant or change the `type` in `mk_dash.py`
> to the element you have. Verify the resource is registered under
> **Settings → Dashboards → (⋮) → Resources**.

After installing, confirm these resources are present (HACS usually adds them
automatically). The two resources the dashboard registers itself are `button-card` and
`card-mod`; the others must exist as **global Lovelace resources**:

```
/hacsfiles/button-card/button-card.js          (module)
/hacsfiles/lovelace-card-mod/card-mod.js       (module)
/hacsfiles/flex-table-card/flex-table-card.js  (module)
/hacsfiles/bubble-card/bubble-card.js          (module)
/hacsfiles/<tabbed-card>/<tabbed-card>.js       (module)
/hacsfiles/kiosk-mode/kiosk-mode.js            (module)
```

> **Kiosk-mode:** install **kiosk-mode** from **HACS → Frontend** (the current default is the
> NemesisRE fork). It must be a **global** Lovelace resource so it loads on the VH-Inventory
> dashboard. The dashboard sets `kiosk_mode.hide_header` to a Jinja template bound to
> `input_boolean.vh_kiosk_mode`, so the header hides only while the Setup-tab toggle is on.

*(Optional)* The default dashboard theme is **Mushroom Square Shadow Hemert** (a custom
theme based on the Mushroom theme). Install the Mushroom theme via HACS if you want the
exact look, or remove/replace the `theme` key in `mk_dash.py`.

**Restart Home Assistant** after installing cards.

## 5. Deploy the configuration package

The package defines all input helpers (text/select/boolean/number) and the scripts that
wire the UI to the backend services.

1. Copy `vh_inventory.yaml` to `/config/packages/vh_inventory.yaml`.
2. Validate: **Developer Tools → YAML → Check Configuration** (or
   `POST /api/config/core/check_config`).
3. Reload: **Developer Tools → YAML → All YAML configuration** (or restart HA).

This creates, among others:

- Input text helpers for product entry and edit IDs (e.g. `input_text.vh_product_name`,
  `input_text.vh_scan_barcode`).
- Input selects for category/store/product/location pickers, the **language selector**
  `input_select.vh_language` (options *English*, *Nederlands*), and the inventory
  **print/filter category** `input_select.vh_print_category`.
- Input numbers for auto-add threshold/quantity and stock quantities.
- One `input_boolean.vh_product_auto_add_enabled`, `input_boolean.vh_show_id_columns`, and
  `input_boolean.vh_kiosk_mode` (drives the *Hide header (kiosk)* toggle).
- Scripts such as `script.vh_save_product`, `script.vh_scan_add`, `script.vh_scan_use`,
  `script.vh_save_resolved_product`, etc.
- Printing scripts `script.vh_print_shopping` (per-store receipt) and
  `script.vh_print_stock` (grouped per category) — these call the
  `escpos_printer.print_text` service and are only functional if the ESC/POS Printer
  integration is installed (see Prerequisites).

## 6. Deploy the backend app

1. Create the folder `/config/pyscript/apps/vh_inventory/`.
2. Copy the backend file into it as `__init__.py`:
   `/config/pyscript/apps/vh_inventory/__init__.py`.
3. Reload pyscript: **Developer Tools → Actions → `pyscript.reload`**, or restart HA.

On first load the app runs `_ensure_schema()`, which **creates the SQLite database and all
tables automatically** at `/config/vh_inventory.db` — no manual SQL needed. The same
routine adds any missing columns on future upgrades (self-healing schema).

4. Confirm the sensors exist (**Developer Tools → States**, filter `vh_inventory`):

   - `sensor.vh_inventory_products`
   - `sensor.vh_inventory_stock`
   - `sensor.vh_inventory_shopping`
   - `sensor.vh_inventory_scanqueue`
   - `sensor.vh_inventory_history`
   - `sensor.vh_inventory_locations`
   - `sensor.vh_inventory_categories`
   - `sensor.vh_inventory_stores`

5. *(Optional)* Force a refresh: **Developer Tools → Actions →
   `pyscript.vh_inventory_refresh`**.

## 7. Build and publish the dashboard

The dashboard is generated by a Python script on your workstation and pushed to HA over
WebSocket. **No file is copied to HA for the dashboard** — it is saved through the
Lovelace API under URL path `/vh-inventory`.

1. On your workstation, install the dependency:

   ```bash
   pip install websocket-client
   ```

2. Place the builder and its translation files together:

   ```
   mk_dash.py
   translations/
     en.json
     nl.json
   ```

   The builder reads `translations/*.json` at build time and embeds them into the
   dashboard for the multi-language layer.

3. Set your connection details as environment variables (the builder reads `HA_HOST` and
   `HA_TOKEN`):

   ```bash
   # Windows PowerShell
   $env:HA_HOST  = "192.168.1.50:8123"
   $env:HA_TOKEN = "<your Long-Lived Access Token>"

   # macOS / Linux
   export HA_HOST="192.168.1.50:8123"
   export HA_TOKEN="<your Long-Lived Access Token>"
   ```

   (Alternatively, edit the `HOST`/`TOKEN` defaults at the top of `mk_dash.py`.)

4. Run it:

   ```bash
   python mk_dash.py
   ```

   Expected output: `save: True`.

5. Open the dashboard at `http://<your-ha>/vh-inventory/main`. You should see the tabbed
   interface (Shopping list, Quick add, Inventory, Products, Locations, Categories, Stores,
   Scan, History, Setup).

## 9. Configuration reference

### External services used (barcode resolution)

The backend resolves unknown barcodes online via these public APIs (requires outbound
internet from HA). The Open\*Facts providers and the UPC Item DB trial are **free and keyless**.
**upcdatabase.org** is optional — supply a free API key to enable it for wider coverage.

| Provider | Base URL | API key |
|---|---|---|
| Open Food Facts | `https://world.openfoodfacts.org` | None — open, keyless |
| Open Beauty Facts | `https://world.openbeautyfacts.org` | None — open, keyless |
| Open Pet Food Facts | `https://world.openpetfoodfacts.org` | None — open, keyless |
| upcdatabase.org | `https://api.upcdatabase.org/product/` | Optional — free key, enables the provider |
| UPC Item DB (trial) | `https://api.upcitemdb.com/prod/trial/lookup` | None, but limited to ~100 lookups/day |

**Resolution order:** Open Food/Beauty/Pet Facts → upcdatabase.org (only when a key is
configured) → UPC Item DB trial. Without a key the app skips upcdatabase.org and still works
using the keyless providers, so a default install needs nothing licensed or stored.

### API keys & secrets (optional)

A default install needs **no keys**. To enable **upcdatabase.org**, register a free account at
<https://www.upcdatabase.org/> to get an API key.

Store the key the Home-Assistant way — never inline it in a package or the app code:

1. Put the secret in `/config/secrets.yaml`:

   ```yaml
   # /config/secrets.yaml
   upcdb_api: "your-key-here"
   ```

2. Hand it to the pyscript app through its config block in `configuration.yaml` (or a
   package), using `!secret` so the value is never written in plain text:

   ```yaml
   # configuration.yaml
   pyscript:
     allow_all_imports: true
     apps:
       vh_inventory:
         upcdb_url_base: https://api.upcdatabase.org/product/
         upcdb_api_key: !secret upcdb_api
   ```

   The app reads its config from `pyscript.app_config`. When `upcdb_api_key` is present the
   backend queries upcdatabase.org after the keyless Open\*Facts providers; when it is absent
   that provider is simply skipped.

`/config/secrets.yaml` is the single, standard location for all Home Assistant secrets; keep
it out of any backup or repository you share.

### Key services (pyscript)

| Service | Purpose |
|---|---|
| `pyscript.vh_inventory_refresh` | Re-publish all sensors from the DB |
| `pyscript.vh_inventory_add_product` | Add a product |
| `pyscript.vh_inventory_scan_enqueue` | Enqueue a scanned barcode (Add/Use) and resolve it |
| `pyscript.vh_inventory_scan_resolve` | Re-run the online lookup for a queued scan |
| `pyscript.vh_inventory_save_resolved` | Save a manually resolved product and finish the scan |
| `pyscript.vh_inventory_add_stock` / `…_adjust_stock` | Add / increment-decrement inventory |
| `pyscript.vh_inventory_add_shopping` / `…_adjust_shop` | Add / adjust shopping-list items |
| `pyscript.vh_inventory_delete` | Delete a row (cascades product → inventory + shopping) |

*(Full lists of helpers, scripts, and services are documented in the product manual and the
package YAML.)*

## 10. Adding a language

1. Copy `translations/nl.json` to `translations/<code>.json` (e.g. `de.json`).
2. Set `__code__` (e.g. `de`) and `__name__` (e.g. `Deutsch`); translate the values.
3. Add the display name to the `input_select.vh_language` options in `vh_inventory.yaml`,
   redeploy the package, and reload input_select.
4. Rebuild the dashboard: `python mk_dash.py`.

> **Convention:** every UI string must be registered in `en.json` + `nl.json` (and any
> other language file). Do not hardcode display text in the builder.

## 11. Troubleshooting

| Symptom | Likely cause / fix |
|---|---|
| `save: False` from the builder | Bad `TOKEN`/`HOST`, or WebSocket blocked. Re-issue the token. |
| Sensors missing | pyscript app not in `/config/pyscript/apps/vh_inventory/__init__.py`, or `pyscript.reload` not run. |
| `ImportError: requests` in logs | `allow_all_imports: true` missing from the pyscript config. |
| Tabs/tables don't render | A custom card resource isn't installed/registered (check Settings → Dashboards → Resources). |
| Language switch does nothing | Hard-refresh the browser (Ctrl+R); confirm `input_select.vh_language` exists. |
| `tabbed-card-programmable` is blank | Install the correct tabbed-card variant or change the `type` in `mk_dash.py`. |

## 12. Backup & restore

Keep a versioned backup containing: the pyscript app, the package YAML, the SQLite DB,
the dashboard builder, and the `translations/` folder. To restore on a new server, repeat
sections 3–7 using the backed-up files. The schema is self-creating, so restoring
`vh_inventory.db` is optional (only needed to restore data).
